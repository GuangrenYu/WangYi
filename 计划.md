# CVE Hunter 多 Agent 与反馈闭环改造计划

日期：2026-06-09

## 目标

当前项目下一步目标是从“串行 PoC 搜索 + 发包 + 防火墙日志判定”升级为“多 agent 分工 + 可替换验证接口 + 自动攻击环境规划 + 反馈闭环”。

本次改造不重写项目，而是在现有 LangGraph 工作流上增量演进。理由是当前项目已经具备 NVD 查询、本地 KB、多源 PoC 搜索、候选池、有限反思、`attempt_history`、IPS 归因、批量重跑和状态码体系。直接重写会丢掉这些已有工程能力，也会让后续验证成本变高。

## 总体布局

保持主入口和现有节点基本不变，把新增能力放到独立模块中：

- `cve_hunter/agents.py`
  - EnvironmentAgent
  - TriggerAgent
  - CriticAgent
  - agent trace 记录

- `cve_hunter/verification.py`
  - RequestExecutor
  - SuccessOracle
  - target-side oracle
  - 兼容当前 `http2pcap` / 内置 `httpx` 发包链路

- `cve_hunter/state.py`
  - 增加 agent 中间产物字段
  - 增加攻击环境字段
  - 增加目标侧 oracle 和复合成功证据字段

- `cve_hunter/graph.py`
  - 只负责工作流编排
  - 不再直接把发包和判定逻辑写死在 `node_verify_poc`

这样后续替换某个 agent 或某种验证方式，不需要重构整个图。

## 工作流设计

目标链路：

```text
CVE 输入
  -> NVD 查询
  -> HTTP/Web 类型判断
  -> EnvironmentAgent 攻击环境规划/可选搭建
  -> 本地 KB 搜索
  -> References 抽取
  -> TriggerAgent 抽象漏洞触发逻辑
  -> 多源 PoC 候选生成
  -> CriticAgent 候选审查/补全
  -> VerifierAgent 发包执行
  -> SuccessOracle 复合成功判定
  -> ReflectionAgent 基于失败证据生成变体
  -> 归档与报告
```

当前已经接入：

- `environment_agent`
- `trigger_agent`
- `critic_agent`
- `verifier_agent`
- `RequestExecutor`
- `SuccessOracle`
- `AGENT_LLM_ENABLED=true` 时，EnvironmentAgent、TriggerAgent、CriticAgent 优先使用 `.env` 中的 DeepSeek 模型；LLM 失败时自动规则降级。

后续可以继续增强：

- IntelAgent：独立整合 NVD、CVE.org、References、nuclei、Exploit-DB、GitHub、搜索结果。
- PoCAgent：从 trigger + environment binding 生成结构化候选。
- ReflectionAgent：从泛化 LLM 反思升级为枚举化动作。
- ReporterAgent：生成更细的证据链和复现报告。

## Agent 职责边界

### EnvironmentAgent

职责：

- 判断当前 CVE 是否能自动搭建攻击环境。
- 查找显式 `ATTACK_ENV_COMPOSE_FILE`。
- 查找本地 `VULHUB_DIR/**/<CVE-ID>/docker-compose.yml`。
- 规划 `target_url`、`target_host`、compose 文件、工作目录。
- 当 `AUTO_ENV_ENABLED=true` 时，执行 `docker compose pull` 和 `docker compose up -d`。

非职责：

- 不生成 PoC。
- 不判断漏洞是否成功。
- 不做任意 shell 执行。

安全策略：

- 默认 `AUTO_ENV_ENABLED=false`，只规划不启动容器。
- 自动执行仅限 docker compose 白名单路径。

相关配置：

```env
AGENT_LLM_ENABLED=true
AGENT_LLM_MODEL=
AUTO_ENV_ENABLED=false
VULHUB_DIR=vulhub
ATTACK_ENV_COMPOSE_FILE=
ATTACK_ENV_TARGET_URL=
CALLBACK_URL=
```

`AGENT_LLM_MODEL` 留空表示复用 `LLM_MODEL`。当前 `.env` 使用：

```env
LLM_BASE_URL=https://api.deepseek.com
LLM_MODEL=deepseek-v4-pro
```

### TriggerAgent

职责：

- 根据 CVE 描述、漏洞类型、受影响产品推断 attack objective。
- 生成 `trigger_candidates`。
- 生成默认 `validation_hint`。

输出示例：

```json
{
  "trigger_id": "cve-xxxx-yyyy-trigger-1",
  "attack_objective": "file_read",
  "trigger_logic": "通过路径或文件参数读取越权文件内容",
  "preconditions": [],
  "variable_slots": ["base_url", "path", "method", "headers", "file_parameter", "file_path"],
  "validation_hint": {
    "type": "response_contains",
    "markers": ["root:", "[boot loader]", "localhost"]
  }
}
```

后续增强方向：

- 从 References、nuclei matcher、Exploit-DB 文本抽取更准确的 trigger。
- 输出多个 trigger candidate，而不是一个轻量推断结果。

### CriticAgent

职责：

- 审查当前 PoC 候选是否可执行。
- 补充 `trigger_id`、`attack_objective`、`validation_hint`、`preconditions`。
- 降低缺证据、缺 Host、格式异常的候选置信度。
- 记录 `candidate_reviews`。

非职责：

- 不发包。
- 不做成败判定。
- 不主动生成新 PoC。

### VerifierAgent

职责：

- 调用 `RequestExecutor` 执行候选。
- 调用 `SuccessOracle` 做复合判定。
- 把执行结果、IPS 结果、target oracle 结果写入 `attempt_history`。

非职责：

- 不判断候选是否合理。
- 不修改 PoC。
- 不搜索情报。

### ReflectionAgent

当前状态：

- 仍复用已有 `node_reflect_after_verify`。
- 已经能基于 HTTP 响应、IPS 结果和最近尝试轨迹生成 0-2 个变体。

后续目标：

- 改成结构化动作：
  - `adjust_path`
  - `adjust_method`
  - `adjust_header`
  - `adjust_content_type`
  - `adjust_payload_encoding`
  - `discover_endpoint`
  - `fetch_csrf`
  - `login_required`
  - `build_multi_step_sequence`
  - `give_up_infrastructure_failure`

## 发包接口设计

统一入口：

```python
class RequestExecutor:
    def execute(self, candidate: dict, environment: dict | None = None) -> dict:
        ...
```

当前默认实现：

- raw HTTP：调用现有 `send_poc_and_capture(raw_http=...)`
- nuclei：调用现有 `send_poc_and_capture(nuclei_yaml=..., target_url=...)`
- request_steps：按步骤顺序执行，先支持简单多请求序列

后续可新增 executor：

- `HttpxExecutor`
- `Http2PcapExecutor`
- `PlaywrightExecutor`
- `SqlmapExecutor`
- `ZapExecutor`
- `BurpExecutor`
- `CallbackAwareExecutor`

原则：

- executor 只负责执行，不负责成功判定。
- executor 返回结构里必须包含 `success`、`status_code`、`body`、`error`、`ips_matches` 等字段。
- 远程 http2pcap 服务失败应归类为基础设施失败，不应直接等价于 PoC 失败。

## 成功判定接口设计

统一入口：

```python
class SuccessOracle:
    def evaluate(self, state, candidate, result, environment=None) -> dict:
        ...
```

当前复合判定顺序：

1. IPS 当前 CVE 命中
   - `status=SUCCESS`
   - `status_code=CAPTURE_SUCCESS`
   - `success_level=ips_cve_match`

2. 目标侧 oracle 成功
   - `status=SUCCESS`
   - `status_code=TARGET_ORACLE_SUCCESS`
   - `success_level=target_oracle`

3. 只有通用/非当前 CVE IPS 命中
   - `status=FAILURE`
   - `status_code=IPS_GENERIC_MATCH_ONLY`
   - 允许进入反思

4. 请求成功但无利用证据
   - `status=FAILURE`
   - `status_code=NO_EXPLOIT_EVIDENCE`
   - 允许进入反思

5. 请求失败
   - `outcome=request_failed`
   - 交给原有错误分类逻辑

保留原则：

- `CAPTURE_SUCCESS` 的语义不变：只有防火墙/IPS 当前 CVE 字段匹配才是该状态。
- 目标侧 oracle 成功是新增成功等级，后续可根据业务需要决定批量统计时是否计入“复现成功”。

## 目标侧 Oracle 设计

当前已支持：

- `response_contains`
- `status_code`
- `nuclei_match`
- `callback`
- `timing`

后续建议支持：

- `sqlmap_result`
- `playwright_dom`
- `auth_state`
- `file_effect`
- `db_effect`
- `service_health`
- `manual_review`

候选结构中应保留：

```json
{
  "attack_objective": "file_read",
  "validation_hint": {
    "type": "response_contains",
    "markers": ["root:"]
  }

```

## 状态字段说明

新增字段：

```python
agent_trace: list[dict]
environment_candidates: list[dict]
attack_environment: dict
trigger_candidates: list[dict]
validation_hints: list[dict]
candidate_reviews: list[dict]
executor_result: dict
oracle_result: dict
target_oracle_success: bool
target_oracle_type: str
target_oracle_details: dict
success_level: str
```

这些字段会写入结果文件，方便后续批量分析。

## 后续改造顺序

### 第一阶段：接口稳定

已完成/本次目标：

- 新增 `agents.py`
- 新增 `verification.py`
- 接入 EnvironmentAgent、TriggerAgent、CriticAgent、VerifierAgent
- 保留原有候选池、反思和 IPS 成功标准
- 补测试

### 第二阶段：PoC 候选升级

目标：

- PoC 候选从 raw request 升级为：

```json
{
  "trigger_id": "...",
  "attack_objective": "...",
  "environment_binding": {},
  "request_steps": [],
  "validation_hint": {},
  "raw_http": "..."
}
```

- 从 nuclei matcher 自动生成 validation hint。
- 从 References 中抽取 trigger candidate。

### 第三阶段：攻击环境增强

目标：

- 支持 vulhub 自动发现。
- 支持 GitHub PoC 仓库中的 compose 文件发现。
- 支持 Docker image 发现和启动。
- 支持健康检查。
- 支持环境清理。

注意：

- 自动执行必须默认关闭。
- 只允许白名单命令。
- 需要记录 compose 文件、启动输出和 target URL。

### 第四阶段：反馈闭环增强

目标：

- 让 ReflectionAgent 输出动作枚举。
- 把失败分类写入 attempt_history。
- 用 target oracle 和 IPS 共同指导下一轮尝试。

### 第五阶段：内部 benchmark

目标：

- 参考 CVE-Bench 建 10-20 个 Docker 化 CVE。
- 每个 CVE 一个 objective 和 grader。
- 用 benchmark 评估多 agent、候选排序、反思是否真的提高成功率。

## 不要做的事

- 不要重写整个项目。
- 不要把所有逻辑塞回 `graph.py`。
- 不要让 LLM agent 直接执行任意 shell 命令。
- 不要放宽 `CAPTURE_SUCCESS` 的含义。
- 不要让基础设施失败污染 PoC 成功率。
- 不要只用 HTTP 200 判断漏洞复现成功。
