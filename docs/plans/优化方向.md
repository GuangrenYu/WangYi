当前是一个单链路 LangGraph 工作流，核心在 graph.py (line 65)：

```
CVE 输入 -> NVD 查询 -> AI 判断是否 HTTP/Web -> 本地 KB -> References 生成 PoC -> nuclei -> Exploit-DB -> imfht -> Web 搜索 + AI 生成 -> 发包验证 -> IPS 命中归因 -> 归档
```

它的特点是：串行回退、每个来源命中后立即验证、失败后进入下一个来源。成功标准比较严格，只有 IPS 日志 CVE 字段明确匹配当前 CVE 才算成功，这一点在 ips_match.py (line 15) 做得不错。

主要问题是：没有反思闭环，没有多 agent 分工，也没有候选池。很多地方只取第一个 PoC 验证，比如 graph.py (line 254)、graph.py (line 324)、graph.py (line 478)。如果第一个路径、协议、header 或 payload 不适配，就直接错过同来源的其他可能。

**方向 1：候选池 + 排序**

| 对比       | 现在                                                         | 改进后                            |
| ---------- | ------------------------------------------------------------ | --------------------------------- |
| PoC 管理   | 每个来源基本取 pocs[0]                                       | 所有来源都放入 poc_candidates     |
| 验证方式   | 命中一个就验证，失败就跳来源                                 | 按可信度排序后逐个验证            |
| 准确率收益 | 容易漏掉第二、第三个候选                                     | 能显著减少“有 PoC 但没试到”的失败 |
| 落地点     | state.py (line 27) 增加候选字段，graph.py (line 226) 各来源改为追加候选 |                                   |

建议新增字段：

python



```
poc_candidates: list[dict] attempt_history: list[dict]
```

每个候选记录 source、raw_http、yaml、evidence_url、confidence、reason。

**方向 2：验证失败后的反思闭环**

| 对比       | 现在                                                         | 改进后                                                 |
| ---------- | ------------------------------------------------------------ | ------------------------------------------------------ |
| 失败处理   | 请求成功但 IPS 未命中，就去下个来源                          | 先分析失败原因，再改写或换候选                         |
| 可利用信号 | 只记录 HTTP 状态码、响应、IPS 摘要                           | 结合状态码、响应片段、generic IPS、错误类型推断下一步  |
| 准确率收益 | 对轻微不适配很敏感                                           | 能修正常见问题，例如协议、路径、Host、header、编码形式 |
| 落地点     | 在 graph.py (line 501) 的 verify_poc 后增加 reflect_after_verify 节点 |                                                        |

比如失败后让反思节点输出结构化动作：

json



```
{  "action": "try_variant",  "reason": "HTTP 404，疑似路径版本不匹配",  "next_candidate": {...} }
```

注意这里不是让模型无限尝试，建议最多 2-3 轮，避免成本失控。

**方向 3：多 agent 分工**

| 对比       | 现在                                   | 改进后                        |
| ---------- | -------------------------------------- | ----------------------------- |
| 架构       | 一个流程里同时做情报、生成、验证、报告 | 多个 agent 各司其职           |
| 质量控制   | 生成后直接验证                         | 生成后先由审查 agent 过滤幻觉 |
| 准确率收益 | AI 可能凭空补路径或 payload            | 证据不足的候选会被降权或丢弃  |
| 落地点     | LangGraph 增加并行/分支节点            |                               |

推荐最小 4 角色：

| Agent         | 职责                                                   |
| ------------- | ------------------------------------------------------ |
| IntelAgent    | 收集 NVD、References、nuclei、Exploit-DB、Web 搜索证据 |
| PoCAgent      | 根据证据生成候选 PoC                                   |
| CriticAgent   | 检查 PoC 是否和 CVE、产品、版本、路径证据一致          |
| VerifierAgent | 发包、抓包、IPS 归因、记录尝试结果                     |

这样不是“多个模型聊天”，而是把现在混在一起的职责拆开，每一步都有可审计输出。

**方向 4：结构化输出替代纯文本正则抽取**

| 对比       | 现在                                    | 改进后                             |
| ---------- | --------------------------------------- | ---------------------------------- |
| LLM 输出   | 直接输出 Raw HTTP                       | 输出 JSON，再由代码渲染 Raw HTTP   |
| 解析方式   | 用正则提取 HTTP 请求                    | 字段化校验 method/path/header/body |
| 准确率收益 | 容易抽错、漏抽、混入解释文本            | 更容易排序、审查、复用、统计       |
| 落地点     | 改 templates.py (line 15) 的 PoC prompt |                                    |

现在 prompt 要求“只输出 HTTP 报文”，但模型仍可能输出解释。改成 JSON 后，CriticAgent 可以检查 evidence_urls 和 assumptions，降低幻觉。

**方向 5：尝试轨迹与批量误差分析**

| 对比       | 现在                                                    | 改进后                                  |
| ---------- | ------------------------------------------------------- | --------------------------------------- |
| 结果记录   | 主要记录最终结果                                        | 记录每个候选的尝试过程                  |
| 批量分析   | 只能看最终通过率、状态码                                | 能看哪个来源、哪类失败、哪类反思最有效  |
| 准确率收益 | 难以知道该优化哪里                                      | 可以用数据反推 prompt、排序和来源优先级 |
| 落地点     | graph.py (line 674) 的 result.json 增加 attempt_history |                                         |
