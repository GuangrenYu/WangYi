# 测试计划 6.10

日期：2026-06-10

## 一、测试目标

本次测试的核心目标不是单纯统计 CVE 复现成功率，而是为后续多 agent、反馈机制、自动攻击环境搭建提供可用样本和失败原因。

重点关注：

- 当前流程是否能稳定跑通：CVE 输入、信息收集、环境分析、PoC 生成、发包、判定、结果落盘。
- 哪些 CVE 因环境缺失无法复现。
- 哪些 CVE 因 PoC、发包方式或前置条件不完整失败。
- 哪些 CVE 已经发包，但防火墙日志无法证明成功。
- 哪些 CVE 可以通过目标侧响应、目标日志或回调服务判断成功。
- 哪些失败样本最适合用于下一步改进 EnvironmentAgent、TriggerAgent、VerifierAgent 和反馈循环。

## 二、测试输入

测试文件位于：

- `F:\wangyi_0\test\A_信息丰富.txt`
- `F:\wangyi_0\test\B_信息中等.txt`

2. `B_信息中等.txt`：抽样测试，用来验证 agent 补全能力。
3. `C_信息较少.txt`：暂不作为主测试集，仅用于后续验证信息不足时系统如何失败。

## 三、执行顺序

### 1. 冒烟测试

先从 `A_信息丰富.txt` 中选取前 30 到 50 个 CVE。

检查：

- 程序是否能连续运行，不因单个 CVE 异常中断。
- `result.json` 是否正常生成。
- agent trace 是否完整。
- 环境候选、触发候选、判定结果是否有输出。
- 防火墙日志查询是否正常。
- 失败原因是否能被记录，而不是只显示 generic failed。

冒烟测试通过后，再跑 A 全量。

### 2. A 组全量测试

A 组是本次主要测试对象。

目标：

- 找出当前流程最容易成功的 CVE。
- 找出有公开 PoC 但失败的 CVE。
- 找出有环境线索但没有自动搭建成功的 CVE。
- 找出发包和判定接口需要改进的样本。

### 3. B 组抽样测试

从 `B_信息中等.txt` 中抽样 50 到 100 个 CVE。

目标：

- 验证信息不完整时 LLM agent 是否能补全环境、PoC、判定线索。
- 观察失败是否集中在环境缺失、PoC 缺失还是判定缺失。
- 和 A 组对比，判断是否需要增强检索或论文/PoC 源。

### 4. C 组暂缓

C 组信息较少，暂时不建议全量跑。

只有在 A、B 流程稳定后，再用 C 组做极端样本测试。

## 五、结果分类标准

### 1. 成功样本

保留以下几类成功样本：

- 防火墙明确命中当前 CVE。
- 防火墙只命中通用攻击规则，但流量确实被识别。
- 防火墙未命中，但目标侧响应可以证明漏洞触发成功。
- 需要回调服务，且 DNSLog/HTTP callback 收到请求。
- nuclei、raw HTTP、curl、脚本等不同发包方式各保留若干个。

成功样本用途：

- 后续作为回归测试集。
- 验证多 agent 改动没有破坏已有成功路径。
- 对比不同判定方式的可靠性。

### 2. 高价值失败样本

这些样本必须重点保留：

- 有公开 PoC，但当前系统没有正确生成或执行。
- 有 Vulhub、Docker、GitHub 环境，但当前系统无法自动搭建。
- 发包已经执行，但防火墙没有日志。
- 目标侧疑似成功，但 verifier 没有识别。
- 需要登录、token、初始化配置、上传文件、特定版本等前置条件。
- 依赖远程回调服务，但当前 callback 接口不足。
- PoC 需要多阶段请求，但当前只发了单个请求。

这些样本最适合用于下一步改进。

### 3. 低价值失败样本

可以记录，但不作为短期重点：

- 漏洞描述过少，找不到利用方式。
- 只存在理论描述，没有公开 PoC、环境或明确触发条件。
- 产品不可获得，版本不可获得，且没有替代镜像。
- 需要商业授权或封闭系统才能测试。

## 六、失败原因分桶

测试完成后，失败 CVE 按以下类别归档：

```text
ENV_MISSING：缺少可用靶场环境
ENV_START_FAILED：环境存在但启动失败
VERSION_MISMATCH：镜像或服务版本不匹配
POC_MISSING：没有可用 PoC
POC_PARSE_FAILED：PoC 存在但无法转成可执行请求
TRIGGER_FAILED：发包失败或请求构造错误
AUTH_REQUIRED：需要登录、token 或权限
PRECONDITION_MISSING：需要初始化数据、配置开关或特定业务状态
CALLBACK_REQUIRED：需要 DNSLog、HTTP callback 或反连服务
FIREWALL_NO_LOG：发包完成但防火墙无日志
FIREWALL_GENERIC_ONLY：只有通用告警，没有当前 CVE 命中
TARGET_ORACLE_MISSING：缺少目标侧成功判定逻辑
TARGET_ORACLE_FAILED：目标侧判定失败
INFRASTRUCTURE_FAILED：网络、代理、端口、Docker、依赖服务失败
```

## 七、agent 改进关注点

### EnvironmentAgent

重点看：

- 是否能识别 Vulhub、Docker Compose、公开镜像。
- 是否能判断目标产品、版本、端口、启动方式。
- 是否能区分环境不存在和环境启动失败。
- 是否能记录需要人工补充的镜像或配置。

后续改进样本：

- 有公开环境但没有识别出来的 CVE。
- 环境识别正确但启动失败的 CVE。
- 需要特定版本但拉错镜像的 CVE。

### TriggerAgent

重点看：

- 是否能从 PoC、nuclei template、curl、Exploit-DB、GitHub 脚本中提取有效请求。
- 是否能处理多阶段请求。
- 是否能识别认证、上传、初始化、回调等前置条件。
- 是否能把发包方式统一转成后续可替换接口。

后续改进样本：

- 有 PoC 但请求生成错误的 CVE。
- 需要多步请求但只执行了一步的 CVE。
- PoC 是脚本但没有转成可控请求的 CVE。

### VerifierAgent

重点看：

- 防火墙命中当前 CVE 时是否正确判定成功。
- 只有通用规则命中时是否单独标记为 traffic detected。
- 防火墙没命中但目标侧成功时是否能识别。
- 是否能记录目标响应、目标日志、callback 结果。

后续改进样本：

- 目标侧已经成功但系统判定失败的 CVE。
- 防火墙日志不充分但有其他证据的 CVE。
- 需要 callback 才能判断成功的 CVE。

### CriticAgent

重点看：

- 是否能提前指出 PoC 缺少参数、认证、版本或初始化步骤。
- 是否能在执行前降低明显错误请求的比例。
- 是否能给出可操作的修复建议，而不是泛泛地说信息不足。

## 十一、优先判断规则

后续筛选时优先保留：

1. 有明确公开 PoC 或公开源码的失败样本。
2. 有 Docker/Vulhub 环境但当前没跑通的样本。
3. 发包完成但防火墙没有命中的样本。
4. 目标侧有成功证据但系统没有判定成功的样本。
5. 需要 callback、认证、多阶段请求的样本。

暂时降低优先级：

1. 没有公开 PoC、没有环境、描述也不清楚的样本。
2. 明显需要商业闭源系统且无法获得测试版本的样本。
3. 只有理论分析、没有可执行触发路径的样本。

## 十二、注意事项

- 不要只保留最终成功率，必须保留失败原因。
- 不要把所有失败都归为无法复现。
- 不要只依赖防火墙日志作为唯一成功标准。
- 防火墙日志、目标响应、目标日志、callback 结果要分开记录。
- 每次修改 agent 后，使用成功样本和高价值失败样本做回归。
- 明天先保证 A 组流程稳定，再扩大测试范围。
