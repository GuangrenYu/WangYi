# 当前项目能力现状总结

生成日期：2026-06-11

项目定位：`CVE Hunter` 当前是一个面向 HTTP/Web 类 CVE 的自动化复现辅助系统。它能完成从 CVE 输入、情报获取、PoC 搜索/生成、候选验证、证据判定、结果归档到批量统计的闭环。它不是完整自主渗透平台，也不是通用靶场生成器；更准确地说，它现在是一个“带安全边界的 CVE 复现工作流引擎”。

## 1. 总体能做到什么程度

当前项目可以做到：

- 输入单个 CVE 或批量 CVE 列表。
- 查询 NVD，提取描述、CVSS、References、受影响产品。
- 用 AI 判断是否属于 HTTP/Web 类漏洞。
- 从多种来源寻找 PoC：本地 KB、NVD References、nuclei、Exploit-DB、imfht、Tavily/DuckDuckGo。
- 基于 References 或搜索结果让 LLM 生成 Raw HTTP PoC 候选。
- 把多个 PoC 统一成候选池，逐个执行或跳过。
- 在执行前检查运行策略，默认 `plan_only` 不真实发包。
- 支持 Raw HTTP、Nuclei YAML、`request_steps` 形式候选的执行接口。
- 通过 IPS 当前 CVE 命中、通用 IPS 命中、目标侧 oracle 进行成功判定。
- 归档 `result.json`、`report.md`、`poc.http`、`poc.yaml`、`environment_manifest.json`。
- 批量跑测试、按状态码重试、核验历史通过记录、统计批量结果。
- 用 milestones 记录流程卡点，例如 NVD、分类、环境、候选、执行、oracle、确认阶段。

当前项目还不能稳定做到：

- 自动从零生成完整可运行靶场。
- 自动修复 Docker compose、初始化应用、创建账号、导入样例数据。
- 自动完成复杂 Web 登录、CSRF、浏览器交互链路。
- 自动执行 sqlmap、ZAP、Burp、Playwright 等外部工具。
- 自动判断所有非 IPS 类漏洞是否真实利用成功。
- 对非 HTTP/Web CVE 做完整复现。
- 对真实公网目标做默认执行；默认策略会阻断发包。

## 2. 攻击环境搭建能力

这是当前最需要讲清楚的部分。现在的环境能力是“发现和启动已有环境”，不是“自动生成任意漏洞环境”。

### 2.1 当前能自动发现的环境

`EnvironmentAgent` 目前能发现两类环境：

1. 显式指定的 compose 文件

配置：

```env
ATTACK_ENV_COMPOSE_FILE=path/to/docker-compose.yml
```

如果这个文件存在，系统会把它作为最高优先级环境候选。

2. 本地 Vulhub 风格目录

配置：

```env
VULHUB_DIR=vulhub
```

系统会在该目录下查找：

```text
**/<CVE-ID>/docker-compose.yml
**/<CVE-ID>/docker-compose.yaml
**/<CVE-ID>/compose.yml
**/<CVE-ID>/compose.yaml
```

例如：

```text
vulhub/weblogic/CVE-2024-0001/docker-compose.yml
```

如果命中，会作为 `vulhub_local` 环境候选。

### 2.2 当前能自动推断什么

对找到的 compose 文件，系统会尝试：

- 解析 `services.*.ports`。
- 取第一个公开端口。
- 推断目标地址，例如：

```text
ports:
  - "18080:80"
```

会推断为：

```text
http://127.0.0.1:18080
```

如果无法从 compose 推断端口，会回退到：

```env
ATTACK_ENV_TARGET_URL
```

或：

```env
TARGET_IP
```

### 2.3 当前能自动启动到什么程度

默认不会启动 Docker。

只有设置：

```env
AUTO_ENV_ENABLED=true
```

并且找到了 compose 候选时，才会执行：

```bash
docker compose -f <compose_file> pull
docker compose -f <compose_file> up -d
```

如果本机没有 `docker compose`，会尝试 `docker-compose`。

启动结果会写入环境信息：

- `setup_result.success`
- `setup_result.commands`
- `setup_result.stdout`
- `setup_result.stderr`
- `setup_result.error`

### 2.4 当前环境 manifest 能记录什么

每次运行会生成：

```text
output/<CVE-ID>/environment_manifest.json
```

里面记录：

```json
{
  "cve_id": "CVE-2024-0001",
  "target_url": "http://127.0.0.1:18080",
  "target_host": "127.0.0.1:18080",
  "source": "vulhub_local",
  "kind": "docker_compose",
  "compose_file": ".../docker-compose.yml",
  "workdir": "...",
  "setup_mode": "disabled",
  "setup_result": {},
  "healthcheck": {
    "url": "http://127.0.0.1:18080",
    "expected_status": 200,
    "timeout_seconds": 60
  },
  "evidence_urls": [],
  "confidence": 0.75,
  "safety": {
    "run_mode": "plan_only",
    "network_scope": "plan_only",
    "allowlist": []
  }
}
```

目前这个 manifest 是“记录和归档”，不是完整环境编排系统。`healthcheck` 字段已经有结构，但当前还没有真正循环探测健康状态。

### 2.5 当前环境搭建不能做什么

现在不能自动完成：

- 从 NVD 描述自动生成 Dockerfile。
- 从 GitHub PoC 仓库自动合成 compose。
- 自动选择正确产品版本镜像。
- 自动初始化数据库、管理员账号、插件、许可证、样例数据。
- 自动执行安装向导。
- 自动等待服务健康并重试修复。
- 自动清理环境。
- 自动搭建多机拓扑、AD 域、内网横向网络、动态防御靶场。

所以当前环境能力的准确表述是：

> 如果本地已经有可用 compose，或者你显式指定了 compose，系统可以发现它、推断目标 URL、可选执行 `docker compose pull/up -d`，并把环境信息归档成 manifest。它还不能从零自动构造任意 CVE 的攻击环境。

## 3. PoC 搜索与生成能力

当前 PoC 获取链路包括：

1. 本地 KB

路径：

```text
poc_kb/custom
poc_kb/trickest-cve
```

能力：

- 优先使用已验证保存的 PoC。
- 从 trickest-cve 关联 GitHub 仓库里尝试抓 README、exploit、poc 文件。
- 从文本里提取 Raw HTTP 请求。

2. NVD References

能力：

- 抓取 NVD references 前若干网页内容。
- 提取正文。
- 把 CVE 描述、受影响产品、网页内容交给 LLM 生成 Raw HTTP PoC 候选。

3. Nuclei

能力：

- 搜索 nuclei 官方模板。
- 如果找到 YAML，则作为 `nuclei_yaml` 候选执行。

4. Exploit-DB / imfht

能力：

- 搜索对应 PoC 来源。
- 尝试提取可执行 Raw HTTP 请求。

5. Web Search

能力：

- 优先 Tavily。
- Tavily 不可用或无结果时回退 DuckDuckGo。
- 搜索结果再交给 LLM 构造 PoC。

当前不能稳定做到：

- 理解任意 Python/Ruby/Go exploit 并自动改写成 HTTP 请求。
- 自动处理多阶段 exploit 的状态保持。
- 自动处理登录态、CSRF token、nonce、动态路径。
- 自动判断 PoC 是否对应当前产品版本。

## 4. Agent 化能力

当前已经有这些 agent 角色：

### EnvironmentAgent

负责：

- 规划目标环境。
- 查找显式 compose。
- 查找本地 Vulhub compose。
- 可选启动 Docker compose。
- 输出攻击环境信息。

不负责：

- 生成 PoC。
- 判断漏洞是否成功。
- 从零生成靶场。

### TriggerAgent

负责：

- 根据 CVE 描述、漏洞类型、产品信息抽象触发逻辑。
- 推断攻击目标类型，例如：
  - `command_execution`
  - `file_read`
  - `database_access`
  - `outbound_callback`
  - `browser_script_execution`
  - `auth_bypass`
  - `denial_of_service`
  - `traffic_detection`
- 生成默认 `validation_hint`。

### CriticAgent

负责：

- 审查当前 PoC 候选是否有可执行 payload。
- 检查 Raw HTTP 格式。
- 检查 Host 头。
- 给候选补充 trigger、preconditions、validation_hint。
- 调整 confidence。

当前 CriticAgent 还比较轻量，主要做格式和元数据审查，还没有做深度语义排序。

### VerifierAgent / SuccessOracle

代码上拆成：

- `RequestExecutor`
- `SuccessOracle`

职责：

- `RequestExecutor` 执行候选。
- `SuccessOracle` 判断执行结果。

这是当前比较清晰的模块边界，后续接 Playwright、sqlmap、ZAP、Burp 时主要应改这里。

## 5. 执行与安全边界

当前默认配置：

```env
RUN_MODE=plan_only
```

这意味着：

- 可以查询情报。
- 可以搜索/生成 PoC 候选。
- 可以规划环境。
- 可以生成报告和 manifest。
- 不会真实发包。

要允许本地靶场执行，需要：

```env
RUN_MODE=local_lab
TARGET_ALLOWLIST=127.0.0.1,localhost,192.168.0.0/16,10.0.0.0/8
```

`local_lab` 会允许：

- `127.0.0.1`
- `localhost`
- 私有网段 IP
- allowlist 命中的目标

要允许明确授权目标，需要：

```env
RUN_MODE=authorized_target
TARGET_ALLOWLIST=example.internal,10.10.10.20
```

`authorized_target` 下，目标必须命中 `TARGET_ALLOWLIST`。

执行预算：

```env
MAX_REQUESTS_PER_CVE=20
MAX_CANDIDATES_PER_CVE=50
```

含义：

- 单个 CVE 最多执行 20 次请求尝试。
- 单个 CVE 最多收集 50 个候选。

## 6. 验证能力

当前支持执行的候选类型：

- Raw HTTP 请求。
- Nuclei YAML。
- 多步 `request_steps`。

执行链路：

- 如果配置 `HTTP2PCAP_URL`，走外部 http2pcap 服务。
- 否则走内置 httpx/scapy 链路。
- PCAP 文件保存在 `output/pcap/`。

成功判定优先级：

1. 当前 CVE IPS 命中

状态：

```text
SUCCESS / CAPTURE_SUCCESS
success_level = ips_cve_match
```

这是最强成功信号。

2. 目标侧 oracle 成功

状态：

```text
SUCCESS / TARGET_ORACLE_SUCCESS
success_level = target_oracle
```

例如响应内容包含预期 marker。

3. 通用 IPS 命中

状态：

```text
FAILURE / IPS_GENERIC_MATCH_ONLY
success_level = generic_ips_only
```

这不算成功，因为没有匹配当前 CVE。

4. HTTP 请求成功但无利用证据

状态：

```text
FAILURE / NO_EXPLOIT_EVIDENCE
success_level = no_exploit_evidence
```

5. 被执行策略拦截

状态：

```text
FAILURE / EXECUTION_POLICY_BLOCKED
success_level = not_executed
```

这通常出现在默认 `plan_only`。

当前 target oracle 支持：

- `response_contains`
- `status_code`
- `nuclei_match`
- `callback`
- `timing`

注意：`callback` 和 `timing` 需要执行器返回对应字段，当前只是 oracle 结构支持，并不等于已经有完整 callback 服务。

## 7. 过程可观测性

当前每次运行会记录 milestones：

```text
input_validated
nvd_loaded
http_classified
environment_planned
environment_ready
trigger_modeled
candidate_collected
candidate_reviewed
request_executed
oracle_evaluated
cve_confirmed
```

这些会写入：

```text
output/<CVE-ID>/result.json
output/batch/*.json
```

作用：

- 不只知道成功/失败。
- 能看到失败卡在哪个阶段。
- 能统计批量任务中最常见瓶颈。

例如：

```json
{
  "candidate_collected": {
    "status": "failed",
    "message": "未收集到新的可执行候选"
  },
  "request_executed": {
    "status": "skipped",
    "message": "RUN_MODE=plan_only blocks network execution"
  }
}
```

## 8. 批量与统计能力

当前支持：

- `python main.py --batch`
- `python main.py --continue`
- `python main.py --classify`
- `python main.py --stats`
- `python main.py --retry-http-failed`
- `python main.py --retry-mode passed`
- `python main.py --retry-mode status --status-code <CODE>`
- 多 VS Code 终端分片执行。

`--stats` 目前能统计：

- JSON 文件数量。
- 完成记录数。
- HTTP / 非 HTTP 数量。
- 正确率。
- 排除非 HTTP 后的 HTTP 正确率。
- 状态码分布。
- PoC 来源分布。
- `success_level` 分布。
- milestone 失败分布。
- milestone 状态分布。

## 9. 当前能力边界总结表

| 能力项 | 当前程度 | 说明 |
|---|---|---|
| CVE 输入 | 可用 | 单个、批量、继续、重试 |
| NVD 情报 | 可用 | 支持 API key，能提取描述、CVSS、References |
| HTTP/Web 分类 | 可用 | LLM 判断，失败时有降级策略 |
| 环境发现 | 初步可用 | 支持显式 compose 和本地 Vulhub compose |
| 环境启动 | 条件可用 | 仅 `AUTO_ENV_ENABLED=true` 且存在 compose 时执行 `pull/up -d` |
| 环境生成 | 不具备 | 不能从零生成 Dockerfile/compose/多机拓扑 |
| 环境 manifest | 可用 | 输出 `environment_manifest.json` |
| 本地 PoC KB | 可用 | 支持 custom 和 trickest-cve |
| 多源 PoC 搜索 | 可用 | References、nuclei、Exploit-DB、imfht、web search |
| LLM PoC 生成 | 初步可用 | 主要生成 Raw HTTP 候选 |
| 候选池 | 初步可用 | 能收集、去重、逐个尝试 |
| 候选排序 | 较弱 | 目前主要按来源流程和 Critic 轻量审查 |
| Raw HTTP 执行 | 可用 | 受 RUN_MODE 策略控制 |
| Nuclei 执行 | 可用 | 通过现有发送/抓包链路 |
| 多步请求 | 接口可用 | 支持 `request_steps`，复杂状态保持仍弱 |
| PCAP 抓包 | 条件可用 | http2pcap 或内置 scapy |
| IPS 成功判定 | 可用 | 区分当前 CVE 命中和通用命中 |
| target oracle | 初步可用 | response/status/timing/callback/nuclei_match 结构支持 |
| 反思修复 | 初步可用 | 失败后 LLM 生成有限变体，但尚未结构化 action 化 |
| 批量统计 | 可用 | 已有 status/source/success_level/milestone 统计 |
| 安全边界 | 可用 | 默认 plan_only，local_lab/authorized_target 可控 |

## 10. 最准确的一句话评价

当前项目已经可以作为 HTTP/Web CVE 自动复现的研究型流水线使用：它能查情报、找 PoC、生成候选、规划已有靶场、受控发包、抓包、判定证据、归档和批量统计。

但它还不是“输入任意 CVE 就自动搭好攻击环境并稳定复现”的系统。攻击环境部分目前做到的是：发现已有 compose、可选启动 compose、推断目标 URL、生成环境 manifest；从零搭建、初始化、健康检查、复杂交互和动态靶场仍是下一阶段工作。
