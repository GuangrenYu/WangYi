# CVE Hunter 本次优化变更总结

生成时间：2026-06-02

## 背景

本次优化围绕当前项目的主要瓶颈展开：历史批量结果中 HTTP 复现正确率偏低，失败集中在 `WEB_SEARCH_FAILED`、`IPS_GENERIC_MATCH_ONLY` 和“找到 PoC 但只验证第一个候选”的路径上。

优化目标不是一次性重构全项目，而是优先提升现有 LangGraph 工作流的命中率、可观测性和可继续迭代能力。

## 一、搜索失败兜底

### 改了什么

- 修改 `cve_hunter/tools/web_search.py`
  - Tavily 配置存在时，先尝试 Tavily。
  - 如果 Tavily 返回错误或没有可用结果，自动 fallback 到 DuckDuckGo。
  - 如果 Tavily 和 DuckDuckGo 都失败，合并两边错误，便于后续归因。

- 修改 `cve_hunter/status_codes.py`
  - 增加对 `usage limit`、`upgrade your plan`、`exceeds your plan` 等套餐/额度错误的识别。
  - 将此类错误归因为 `API_QUOTA_EXHAUSTED`。

- 新增/修改测试
  - `tests/test_web_search.py`
  - `tests/test_status_codes.py`
  - `tests/__init__.py`

### 为什么要改

历史批量结果中 `WEB_SEARCH_FAILED` 有 576 条，且全部来自 Tavily 套餐/额度限制。原逻辑只要配置了 Tavily，就不会走 DuckDuckGo，导致搜索链路被单点额度问题阻断。

改完后，Tavily 额度耗尽不再直接终止搜索，而是自动尝试 DuckDuckGo，提高 PoC 搜索阶段的可用性。

## 二、候选池与尝试轨迹

### 改了什么

- 修改 `cve_hunter/state.py`
  - 新增 `poc_candidates`
  - 新增 `current_candidate_index`
  - 新增 `attempt_history`

- 修改 `cve_hunter/graph.py`
  - 各 PoC 来源不再只取 `pocs[0]`。
  - Reference、Nuclei、Exploit-DB、imfht、Web Search、本地 KB 产出的 PoC 会进入统一候选池。
  - 验证失败后，优先尝试候选池里的下一个 PoC。
  - 每次验证都会写入 `attempt_history`。
  - 最终 `result.json` 会写入 `poc_candidates`、`current_candidate_index`、`attempt_history`。

- 新增测试
  - `tests/test_graph_candidates.py`

### 为什么要改

原工作流虽然某些来源能提取多个 PoC，但实际验证时只使用第一个。典型位置包括：

- Reference AI 生成后只用第一个
- Exploit-DB 提取后只用第一个
- imfht 提取后只用第一个
- Web Search AI 生成后只用第一个

这会造成明显漏试：第一个路径、header、payload 不适配时，后续候选完全没有机会。

候选池解决的是“有 PoC 但没试全”的问题；`attempt_history` 解决的是“失败后不知道失败在哪里”的问题。

## 三、JSON-first PoC 输出与公共解析器

### 改了什么

- 新增 `cve_hunter/poc_parser.py`
  - `parse_poc_candidates_json()`
  - `render_raw_http()`
  - `extract_http_requests()`

- 修改 `cve_hunter/prompts/templates.py`
  - `POC_GENERATION_FROM_REFS` 改为要求输出 JSON candidates。
  - `POC_GENERATION_FROM_SEARCH` 改为要求输出 JSON candidates。

- 修改 `cve_hunter/graph.py`
  - LLM 输出优先按 JSON 解析。
  - JSON 解析失败时，再 fallback 到旧 Raw HTTP 提取。

- 修改 `cve_hunter/tools/local_kb.py`
  - 本地 KB / GitHub 抓取统一使用公共 Raw HTTP parser。

- 新增测试
  - `tests/test_poc_parser.py`

### 为什么要改

旧 Prompt 要求模型直接输出 Raw HTTP，但模型仍可能混入解释文本、多个请求粘连、Markdown 包裹或不稳定分隔符。这样后续候选池、排序、审查和反思都缺少稳定字段。

JSON-first 的好处：

- method、path、headers、body 字段可校验
- evidence_url、confidence、reason 可用于后续排序和审查
- 程序统一渲染 Raw HTTP，减少格式漂移
- 旧网页和本地 KB 仍通过 Raw HTTP fallback 兼容

## 四、有限反思闭环

### 改了什么

- 修改 `cve_hunter/state.py`
  - 新增 `reflection_rounds`
  - 新增 `max_reflection_rounds`，默认 2

- 修改 `cve_hunter/prompts/templates.py`
  - 新增 `POC_REFLECTION_AFTER_VERIFY`

- 修改 `cve_hunter/graph.py`
  - 新增 `node_reflect_after_verify`
  - 验证失败后的流转变为：
    1. 先尝试候选池下一个候选
    2. 候选耗尽后，如果满足条件，进入反思节点
    3. 反思生成变体候选后回到验证
    4. 反思没有有效候选或超过轮次，再进入下一个来源

### 反思触发条件

当前只在以下场景触发反思：

- HTTP 请求成功，但当前 CVE IPS 未命中
- 出现通用/非当前 CVE IPS 命中

当前不会在以下场景触发反思：

- 目标连接失败
- http2pcap 服务失败
- 基础设施/网络类请求失败

### 为什么要改

`IPS_GENERIC_MATCH_ONLY` 在历史结果中有 401 条，说明 PoC 经常能触发一些规则，但没有匹配当前 CVE。这类失败通常可能由 path、header、method、参数、Content-Type 或编码细节不适配导致。

反思节点不是让模型无限重试，而是在最多 2 轮内生成小变体，限制修改范围，避免成本失控和幻觉扩大。

## 七、预期收益

### 短期收益

- Tavily 额度问题不再直接导致搜索失败。
- 多个 PoC 候选会被实际验证，而不是只验证第一个。
- 每次验证过程可审计，便于后续分析。
- LLM PoC 输出更结构化，降低解析不稳定。
- 对轻微不适配的 PoC，反思节点能生成有限变体。

### 中期收益

这些改动为后续优化打下基础：

- 候选排序
- Critic 审查
- 基于历史成功率的来源权重
- 批量误差分析
- 多 agent 分工

## 八、建议的下一步

下一步不建议继续堆功能，建议先做小规模效果评估。

推荐重跑两类历史失败样本：

```powershell
python main.py --retry-mode status --status-code IPS_GENERIC_MATCH_ONLY --start 1 --end 50
python main.py --retry-mode status --status-code WEB_SEARCH_FAILED --start 1 --end 50
```

观察指标：

- `CAPTURE_SUCCESS` 是否增加
- `attempt_history` 是否记录多个候选
- `reflection_rounds` 是否触发
- 反思候选是否大量重复或无效
- `WEB_SEARCH_FAILED` 是否因 DuckDuckGo fallback 减少

如果小规模结果有效，下一步再做候选排序：

- `local_kb_custom` 最高优先级
- `nuclei` 次高
- 有 `evidence_url` 的候选优先
- `confidence` 高的候选优先
- 历史成功率高的来源优先

## 九、注意事项

- 本次改动没有放宽成功标准，仍然只有当前 CVE IPS 命中才算成功。
- 反思节点不会处理基础设施失败，避免浪费 LLM 成本。
- Retry 会覆盖历史 `output/batch` JSON，执行前建议备份结果目录。
- 当前工作区仍存在 `poc_kb/trickest-cve` 子模块状态变更和未跟踪的 `优化方向.md`，提交前需要单独确认。
